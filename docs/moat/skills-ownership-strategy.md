# SkillsTech moat strategy

The moat should not be created by giving every product 50 hours and releasing many disconnected applications. That creates breadth, maintenance cost, and duplicated features.

The 30-day objective should be:

> **Ship one complete Skills Ownership Loop across four core products, then use Certified and GetSkillsTech to distribute and monetize it.**

The loop:

```text
Requirement or intended change
        ↓
OpenThunder verifies the implementation
        ↓
Repo Mastery teaches what changed
        ↓
Skills Tech Talk verifies human understanding
        ↓
Workspace issues portable Ownership Proof
        ↓
Certified and Communities turn proof into opportunity
```

SkillsTech becomes increasingly defensible each time this loop is completed because the company accumulates structured evidence connecting:

* What was intended
* What was built
* Whether it was verified
* What the person understands
* What the person can explain
* What skills were demonstrated
* What evidence supports those skills
* How mastery changes over time

## The central proprietary asset

# The seven layers of the moat

| Layer           | What SkillsTech builds                                                    | Why it becomes defensible                                  |
| --------------- | ------------------------------------------------------------------------- | ---------------------------------------------------------- |
| Workflow        | Verification, learning, defense, and proof in one flow                    | Competitors must reproduce the entire workflow             |
| Data            | Ownership Graph linking skills, work, evidence, and outcomes              | Becomes more valuable after every completed activity       |
| Trust           | Local-first verification, BYOK, reproducible checks, transparent evidence | Enterprise customers trust SkillsTech with sensitive work  |
| Learning        | Adaptive mastery, retention, drills, explain-back, and reassessment       | Personalization improves with usage history                |
| Switching costs | Team policies, mastery history, architecture maps, proof records          | Teams lose valuable operational context when leaving       |
| Network         | Communities based on demonstrated skills and verified contributions       | Reputation becomes more useful as participation grows      |
| Protocol        | Intent and Proof schemas that other tools can produce or consume          | SkillsTech becomes infrastructure, not just an application |

# What Communities becomes

Communities should replace Social, Talent, and Connect, but it should not become a generic developer social network.

Its positioning should be:

> **The professional community where reputation is supported by demonstrated skills, verified work, and meaningful contributions.**

Communities eventually includes:

* Proof-backed professional profiles
* Skills and technology communities
* Verified project and repository contributions
* Public learning milestones
* Company communities
* Expert help and office hours
* Mentorship
* Jobs and talent discovery
* Recruiter and employer access
* Technical discussions tied to evidence
* Courses and paid knowledge
* Community reputation based on contribution quality

The network moat comes later. Communities needs meaningful proof before it can become defensible.

Do not build a full feed during the first 30 days. Build a proof profile, proof sharing, and a Communities waitlist.

## Days 8 through 14: Release OpenThunder ownership verification

### Primary focus

**AI Change Ownership Check**

The narrow workflow:

1. Select a repository, branch, commit, or pull request.
2. Identify the changed files and affected components.
3. Run deterministic architecture, quality, security, and test checks.
4. Explain what changed.
5. Identify what could break.
6. Identify what the responsible engineer should understand.
7. Produce normalized `VerificationEvidence`.
8. Send that evidence to Workspace.

### Required outputs

* Change summary
* Architecture impact
* Security findings
* Quality findings
* Test coverage and gaps
* Dependency impact
* Risk level
* Unresolved questions
* Evidence references
* Reproducible command or run identifier

### Trust requirements

* Deterministic checks before AI interpretation
* AI clearly marked as advisory
* Local execution wherever practical
* No source-code retention by default
* BYOK support where AI is used
* Hashes, metadata, and summaries stored instead of source code
* Clear export and deletion options

### Depth limit

Do not add ten new lenses. Make one complete ownership verification workflow excellent.

### Release target

**Day 14: OpenThunder Local Ownership Check v1**

## Days 15 through 21: Complete the human ownership loop

This week joins Repo Mastery and Skills Tech Talk.

## Repo Mastery scope

Consume the repository, git diff, and OpenThunder evidence.

Generate a **Change Mastery Pack** containing:

* What changed
* Why it changed
* Affected architecture
* Important files and symbols
* Data flow
* Security implications
* Failure scenarios
* Testing strategy
* Terms and glossary
* Five essential questions
* Flashcards
* A short mastery quiz
* Areas the user still needs to inspect

Every claim should reference evidence from the repository or OpenThunder.

Produce normalized `MasteryEvidence`.

## Skills Tech Talk scope

Create one excellent room:

**Code Review Defense**

The experience should:

1. Read the change and its evidence.
2. Ask the engineer to explain the purpose.
3. Ask how the implementation works.
4. Ask what could fail.
5. Ask about security and architecture.
6. Ask how the change was tested.
7. Challenge weak or vague explanations.
8. Score the response using a transparent rubric.
9. Produce normalized `DefenseEvidence`.
10. Recommend targeted Repo Mastery practice.

Support spoken and written defense when practical, but do not delay the release for advanced voice capabilities.

### End-to-end definition of done

A user can:

```text
Verify a change in OpenThunder
        ↓
Study it in Repo Mastery
        ↓
Defend it in Skills Tech Talk
        ↓
Receive an Ownership Proof in Workspace
```

# Depth rules

Use four implementation levels.

## Level 1: Link integration

Products share navigation and branding.

This is not a moat.

## Level 2: Identity integration

Products share users, organizations, and subscriptions.

Necessary, but not sufficient.

## Level 3: Evidence integration

One product produces evidence that another product consumes.

This creates a workflow moat.

## Level 4: Adaptive closed loop

Evidence from each product changes what the other products recommend, teach, verify, and assess.

This creates the strongest moat.

### Thirty-day targets

| Product relationship             |      Required level |
| -------------------------------- | ------------------: |
| OpenThunder to Workspace         |             Level 3 |
| OpenThunder to Repo Mastery      |             Level 3 |
| Repo Mastery to Skills Tech Talk |             Level 3 |
| Skills Tech Talk to Workspace    |             Level 3 |
| Certified to Skills Tech Talk    |             Level 3 |
| Workspace to all products        | Level 2 and Level 3 |
| Mobile to mastery system         |             Level 2 |
| Communities to proof system      |             Level 2 |
| ThunderLang to evidence model     |    Internal Level 3 |
| Runtime to proof system          |         Design only |

Level 4 becomes the focus after the initial release.

# Canonical ownership rules

- Workspace owns shared identity, organizations, entitlements, evidence, proof, and audit history.
- OpenThunder owns deterministic software verification evidence.
- Repo Mastery owns repository and change understanding.
- Skills Tech Talk owns demonstrated human explanation and defense.
- Skills Tech Certified owns standardized learning, assessment, and exam-readiness evidence.
- Communities owns proof-backed identity, contribution, connection, opportunity, and reputation.
- SkillsTech Mobile owns mobile practice, retention, notifications, and proof access.
- SkillsTech IDE owns the integrated local development surface.
- SkillsTech Cloud owns hosted collaboration, team policies, synchronization, and organization processing.
- ThunderLang owns the semantic Intent and Proof model.
- SkillsTech Runtime owns runtime enforcement and production evidence.
- GetSkillsTech owns umbrella positioning, product discovery, onboarding, and distribution.

Shared evidence events should support at least:

- verification.completed
- mastery.generated
- mastery.assessed
- defense.completed
- certification.assessed
- credential.issued
- proof.issued
- proof.revoked
- skill.demonstrated
- skill.reassessed

All evidence must support:

- Stable identifiers
- Schema versioning
- Evidence references
- Privacy classification
- Visibility controls
- Source product
- User and organization ownership
- Creation timestamp
- Reproducibility where applicable

Trust principles:

- Local-first when practical
- Deterministic before generative
- BYOK where AI is optional
- Do not retain customer source code by default
- Store hashes, references, metadata, and necessary summaries
- Clearly distinguish evidence from AI interpretation
- Make exported evidence understandable
- Never imply that a SkillsTech training credential is an official provider certification

# ThunderLang's role in the moat (this repo)

ThunderLang is **Tier 3: integrate lightly, internal schema only**. It is the Protocol
layer of the moat (layer 7) and the semantic Intent and Proof model. During the first
30 days ThunderLang does **not** try to finish a general-purpose language, and it does
**not** compete with the core Tier-1 release for attention.

ThunderLang's moat obligations:

- **Protocol / layer 7.** Own the semantic Intent and Proof model that other products
  can produce or consume: what should happen, what must never happen, what constraints
  apply, what evidence verifies the intent, what skills are required, what human
  understanding must be demonstrated.
- **ThunderLang to evidence model = Internal Level 3.** Provide adapters/examples showing
  how an Intent record connects to OpenThunder `VerificationEvidence`, Repo Mastery
  content, Skills Tech Talk defense questions, and the Workspace `ProofBundle`, without
  requiring customers to rewrite software in ThunderLang.
- **Every proof shows its intent.** The Ownership Loop must be able to reference a stable
  Intent model, so every proof can show which intent or requirement the evidence supports.
- Prioritize interoperability, stable identity, source mapping, and deterministic
  verification. Do not require rewrites. See the actionable checklist in `TODO.md`
  (section: "SkillsTech moat , ThunderLang obligations").
